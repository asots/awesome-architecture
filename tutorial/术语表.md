# 附录 · 架构术语表

> 读教程或模板时撞到不认识的词?来这里查。每条只给**一句话直觉** + 指向讲透它的章节。
> 不求严谨定义,只求你「秒懂它在说什么」。

---

## 规模与性能

| 术语 | 一句话直觉 | 讲透它的地方 |
|---|---|---|
| **QPS / TPS** | 每秒请求数 / 每秒事务数——衡量系统多忙。 | [07 信封背面估算](07-从0到1设计一个系统.md) |
| **延迟(Latency)** | 单次操作要等多久(快不快)。 | [06 质量属性](06-质量属性与取舍.md) |
| **吞吐(Throughput)** | 单位时间能处理多少(量大不大)。 | [06](06-质量属性与取舍.md) |
| **P99(尾延迟)** | 99% 的请求比这个值快——看 P99 别看平均,因为最慢的那 1% 最伤体验。 | [06](06-质量属性与取舍.md) |
| **读写比** | 读多还是写多——决定系统该往「读优化」还是「写优化」倾斜。 | [02 灵魂六问](02-架构师的思考框架.md) |
| **信封背面估算** | 用几个除法快速估出数量级(QPS、存储量),判断系统会被什么压垮。 | [07](07-从0到1设计一个系统.md) |

## 数据与一致性

| 术语 | 一句话直觉 | 讲透它的地方 |
|---|---|---|
| **强一致 / 最终一致** | 写完立刻全网都对(贵)/ 写完过一会儿才一致(便宜、高可用)。 | [05 数据与状态](05-数据与状态.md) |
| **CAP** | 网络一旦分区,只能在「一致」和「可用」里二选一。 | [05](05-数据与状态.md) |
| **ACID / BASE** | 严谨派(强一致、强保证)/ 务实派(高可用、最终一致)。 | [05](05-数据与状态.md) |
| **事务(Transaction)** | 一组操作「要么全成、要么全不成」。 | [05](05-数据与状态.md) |
| **幂等(Idempotency)** | 同一个操作重复执行多次,结果和执行一次一样——重试不会闯祸。 | [支付系统模板](../templates/payment-system/README.md) |
| **无状态 / 有状态** | 不记事(好复制好扩)/ 记事(难复制,一切麻烦的根)。 | [05](05-数据与状态.md) |

## 扩展手段

| 术语 | 一句话直觉 | 讲透它的地方 |
|---|---|---|
| **垂直扩展 / 水平扩展** | 把单机搞更强(治标、有上限)/ 加更多机器(治本、要求无状态)。 | [06](06-质量属性与取舍.md) |
| **复制(Replication)** | 做多份只读副本,主要为「扩读」。 | [05](05-数据与状态.md) |
| **分片(Sharding)** | 按规则把数据切到多台机器,主要为「扩写」。 | [05](05-数据与状态.md) |
| **缓存(Cache)** | 把热点数据放在更快的地方,降延迟 + 扩读;代价是一致性。 | [05](05-数据与状态.md) |
| **CDN** | 把内容铺到离用户最近的边缘节点,降延迟 + 省带宽。 | [视频流媒体模板](../templates/video-streaming/README.md) |
| **削峰 / 限流** | 用队列 / 排队把瞬时洪峰整形成平缓水流,别让后端被冲垮。 | [04 消息队列](04-十大核心架构模式.md) |

## 架构模式

| 术语 | 一句话直觉 | 讲透它的地方 |
|---|---|---|
| **单体 / 微服务** | 一个部署单元(简单,被低估)/ 多个独立部署的小服务(解决「人」的扩展,被滥用)。 | [04](04-十大核心架构模式.md) |
| **事件驱动 / Pub-Sub** | 广播「发生了什么」,谁关心谁响应——解耦与扇出。 | [04](04-十大核心架构模式.md) |
| **CQRS** | 把「写」和「读」拆成两套模型各自优化。 | [04](04-十大核心架构模式.md) |
| **扇出(Fan-out)** | 一件事触发很多下游——推模型(写时扇出)vs 拉模型(读时聚合)。 | [社交信息流模板](../templates/social-feed/README.md) |
| **倒排索引** | 把「文档→词」翻转成「词→文档列表」,全文检索的根基。 | [搜索引擎模板](../templates/search-engine/README.md) |
| **ANN(近似最近邻)** | 用一点精度换巨大速度,在海量向量里快速找「最相似」。 | [向量数据库模板](../templates/vector-database/README.md) |

## 可靠性与运维

| 术语 | 一句话直觉 | 讲透它的地方 |
|---|---|---|
| **可用性 / 几个 9** | 系统有多少时间是「活着」的;99.9% = 3 个 9,每多一个 9 成本数量级上涨。 | [06](06-质量属性与取舍.md) |
| **持久性(Durability)** | 存进去的数据丢失的概率有多低(如 S3 的 11 个 9)。 | [06](06-质量属性与取舍.md) |
| **SLO / 错误预算** | 给可用性定个目标(SLO),剩下的「不可用额度」就是错误预算,用完就停新功能保稳定。 | [06](06-质量属性与取舍.md) |
| **单点故障(SPOF)** | 某一处一挂全系统瘫——可用性的头号敌人,靠冗余消灭。 | [06](06-质量属性与取舍.md) |

## 流程与演进

| 术语 | 一句话直觉 | 讲透它的地方 |
|---|---|---|
| **质量属性 / 非功能性需求** | 系统「做得多好」(快/稳/扛得住),区别于「做什么」(功能)。 | [02](02-架构师的思考框架.md) |
| **取舍(Trade-off)** | 任何决策都是「用 A 换 B」;没有银弹。 | [02](02-架构师的思考框架.md) |
| **ADR** | 架构决策记录:用一页纸记下「为什么这么决定、放弃了什么」。 | [08 ADR](08-架构决策记录与演进.md) |
| **技术债** | 为「现在更快」有意识地选的权宜方案——关键是记账、按计划还。 | [08](08-架构决策记录与演进.md) |
| **康威定律** | 系统架构会长得像设计它的组织的沟通结构。 | [08](08-架构决策记录与演进.md) |
| **C4 模型** | 像地图缩放一样分四层画架构图:Context→Container→Component→Code。 | [03 画架构图](03-读懂与画好架构图.md) |
